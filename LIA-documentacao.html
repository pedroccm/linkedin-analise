Visão geral

LIA é um SaaS para rastrear e analisar perfis (pessoas e empresas) do LinkedIn. O usuário adiciona um perfil pela URL, o LIA puxa via Apify os posts, reações, comentários e funcionários, armazena no Supabase e apresenta em feed, timeline e páginas de perfil. Tem auth por email/senha, billing por PIX, planos por número de perfis, workspaces corporativos compartilhados e monitoramento diário com alerta de WhatsApp.

   -> /api/profiles/[id]/sync           chama Apify (posts + detalhes)

Multi-tenant & planos

Cada dado pertence a um workspace. Para um usuário solo, o workspace é ele mesmo. Para um membro de organização, o workspace é o dono da org: todos os membros compartilham os mesmos perfis, tags e feed.

Como funciona o workspace

  • Leitura (RLS): as tabelas linkedin_* filtram por user_id in (select lia_workspace_ids()). A função SQL lia_workspace_ids() retorna o próprio auth.uid() mais os owner_id de toda org em que o usuário é membro.
  • Escrita: workspaceOwnerId(supabase, userId) (em lib/workspace.ts) resolve o dono do workspace; os dados são gravados sob o id dele.
  • Assinatura: getSubscriptionStatus (em lib/subscription.ts) resolve plano e contagem de perfis pelo dono do workspace, então membros herdam o plano da org.

app/page.tsx · app/actions.ts

  • Deslogado: renderiza a landing (hero, features, preços, FAQ). Logado: form de adicionar perfil + lista.
  • Lê linkedin_profiles; filtro por tipo (todos / pessoas / empresas) via ProfileList.
  • Sugestões de perfis (top voices) aparecem quando há menos de 2 perfis (lib/suggested-profiles.ts).
  • SyncAllButton dispara o sync de todos os perfis em segundo plano.

Server actions (app/actions.ts)

  • addProfile: normaliza a URL, detecta tipo (in/company), checa canAddProfile (limite do plano, senão /billing?reason=limit), faz upsert deduplicado por (workspace, url) e redireciona para /profiles/{id}?autosync=1.
  • trackEmployee: transforma um funcionário de empresa em perfil rastreado e faz o back-link em linkedin_company_employees.tracked_profile_id.
  • deleteProfile: remove o perfil.

/profiles/[id]

app/profiles/[id]/page.tsx

  • Aceita UUID ou handle na URL (ex.: /profiles/williamhgates).
  • Cabeçalho: avatar (espelhado no Storage), chips (verificado, criador, influencer, premium, etc.), métricas, tags, botão Monitorar e botão Sync.
  • Abas para pessoa: Posts, Reactions, Comments. Para empresa: Feed, Timeline, Posts, Employees.
  • Posts têm estatísticas (StatsCard, posts por semana, top autores engajados) e filtros client-side instantâneos.
  • Tags: criar/atrelar tags ao perfil (app/tag-actions.ts).

  • Mostra plano atual + perfis usados/limite + dias restantes.

APIs / rotas

Todas em app/api/**/route.ts. As rotas de sync rodam no runtime Node com maxDuration = 300.

Perfil · sincronização

POST /api/profiles/[id]/sync login

Sincroniza detalhes + posts (pessoa ou empresa). Retorna { inserted, total, detailsLoaded?, errors? }.

POST /api/profiles/[id]/sync-reactions login

Sincroniza as reações (likes) do perfil. Retorna { inserted, total }.

POST /api/profiles/[id]/sync-comments login

Sincroniza os comentários do perfil. Retorna { inserted, total }.

POST /api/profiles/[id]/sync-employees login

Sincroniza funcionários (só empresa). Retorna { inserted, total }.

Monitoramento

GET POST /api/cron/monitor CRON_SECRET

Roda o motor de monitoramento. Autorização por header Authorization: Bearer <CRON_SECRET> ou query ?secret=<CRON_SECRET>. Retorna { ok, checked, newPosts, events, errors }. Veja a seção Monitoramento.

Pagamento

POST /api/checkout login

Cria cobrança PIX via AbacatePay. Body: { plan?: "starter"|"pro"|"corporate" }. Insere em linkedin_payments (status pending) e retorna { paymentId, pixId, brCode, brCodeBase64, amount, expiresAt }. O PIX vale 30 min.

GET /api/checkout/status login

Consulta o status do PIX (?paymentId=...). Quando pago, executa fulfillPayment(): marca como paid e estende subscription_until em +30 dias. ?simulate=true só funciona em dev.

Diversos

POST /api/locale público

Define o cookie lia_locale (1 ano). Se logado, persiste em linkedin_users_meta.locale. Body: { locale: "pt"|"en" }.

POST /api/admin/users/[userId]/sync-all admin

Sincroniza todos os perfis de um usuário (details+posts; empresa: employees; pessoa: reactions+comments). Retorna { profiles, steps, ok, failed, results[] }.

Cache global de scrape

tabela linkedin_scrape_cache · função cachedActor() em lib/sync-core.ts

  • Chave: (canonical_url, sync_type). A URL é normalizada por canonicalLinkedinUrl() para https://www.linkedin.com/{in|company}/{handle}.
  • TTL: SCRAPE_CACHE_TTL_HOURS (24h por padrão). Dentro do TTL, qualquer workspace reusa o resultado em vez de pagar o Apify de novo.
  • Só service-role lê/escreve (RLS ligada, sem policies), então não vaza entre usuários.

Monitoramento

lib/monitor.ts · app/api/cron/monitor/route.ts · app/profiles/[id]/monitor-*.*

O LIA detecta posts novos e enfileira eventos de alerta. O envio do WhatsApp é um sistema à parte (a construir) que consome a fila. O LIA nunca envia mensagem.

Como funciona

  1. Toggle Monitorar 🔔 na página do perfil grava monitor_enabled (compartilhado no workspace). Ao ligar, usa o último post conhecido como baseline (não alerta sobre o que já existia).
  2. O número de WhatsApp fica por usuário em /account (linkedin_users_meta.whatsapp).
  3. Um cron diário bate em /api/cron/monitor. O motor faz um probe do post mais recente de cada perfil monitorado, reusando o cache global (perfil monitorado por vários workspaces custa 1 chamada Apify).
  4. Se o post mudou, enfileira 1 linha por destinatário (todos no workspace com WhatsApp) em linkedin_monitor_events e atualiza last_monitored_post_id.
O agendador ainda não está definido. Basta bater no endpoint 1x/dia (n8n, Netlify scheduled function, etc.) com o CRON_SECRET que está nas env vars do Netlify.

Contrato da fila (linkedin_monitor_events)

RLS ligada sem policies: só service-role escreve; o sender externo lê via service-role ou conexão direta no Postgres.

ColunaConteúdo
profile_name, profile_urlQuem publicou
apify_post_id, post_url, post_excerpt, posted_atO post novo
recipient_user_id, whatsappPara quem mandar
statuspending | sent
created_at, sent_atCarimbos

Idempotência: unique (profile_id, apify_post_id, recipient_user_id).

O que o sistema de envio deve fazer

select * from linkedin_monitor_events where status = 'pending';
-- montar a mensagem com profile_name + post_excerpt + post_url
-- enviar para a coluna whatsapp
update linkedin_monitor_events
   set status = 'sent', sent_at = now()
 where id = ...;

Função e provisionamento

  • lia_workspace_ids(): SET de user_ids visíveis (próprio + donos das orgs em que é membro). Base de toda RLS por workspace.
  • Provisionamento do usuário do LIA é lazy via ensureLiaUser() (lib/ensure-lia-user.ts): upsert em app_users e linkedin_users_meta no primeiro acesso (a base é compartilhada com o fotofocinho, então não há trigger global de signup só pro LIA).
  • Storage: bucket público avatars guarda cópias dos avatares/logos (as URLs do LinkedIn expiram e bloqueiam hotlink).

Rodar local

O ambiente local usa Supabase via Docker. Scripts em package.json.

# 1. instalar dependências (rodar você mesmo; o fluxo não roda npm install automático)
npm install

# 2. subir o Supabase local (precisa do Docker Desktop)
npm run db:start        # API 54331 · DB 54332 · Studio 54333

# 3. aplicar todas as migrations no banco local
npm run db:reset

# 4. subir o dev server
npm run dev             # http://localhost:3000

# parar o Supabase
npm run db:stop
  • Config local em supabase/config.toml (project_id "linkedin", Postgres 15).
  • Studio do Supabase (UI do banco): http://127.0.0.1:54333.
  • O .env.local aponta para o Supabase local por padrão.

Deploy & migrations

Deploy (Netlify)

  • Site: lia-analise (o subdomínio não pode conter "linkedin", marca registrada).
  • Repo: pedroccm/linkedin-analise, branch master. Deploy automático a cada push (CI do Netlify).
  • Produção usa a base Supabase da nuvem (compartilhada com o fotofocinho), com as env vars definidas no painel do Netlify.

Aplicar migration na nuvem

Via Supabase CLI com a connection string da base (a senha fica fora deste doc):

npx supabase db push --db-url \
  "postgresql://postgres:<SENHA>@db.iynirubuonhsnxzzmrry.supabase.co:5432/postgres"
Como a base é compartilhada com o fotofocinho, o histórico remoto contém migrations do fotofocinho. Para o push não reclamar de "remote migration not found in local", os arquivos de migration do fotofocinho (timestamps 20260301*) precisam estar copiados em supabase/migrations/ do LIA. O push aplica só as migrations novas.

Área admin

app/admin/* · lib/admin.ts

Acesso restrito por isAdminEmail(), que compara o email contra a allowlist ADMIN_EMAILS. A proteção é em três camadas: no layout (/admin/** retorna 404 para não-admin), nas server actions (requireAdmin()) e em rechecagem na página de uso.

Server actions (app/admin/actions.ts)

  • extendSubscription: estende subscription_until (a partir da data atual da assinatura, ou de agora). Se o usuário estava free, assume Starter.
  • setPlanTier: muda o plano; se free, limpa a validade.
